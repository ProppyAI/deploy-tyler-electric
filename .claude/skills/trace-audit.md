# /harness:trace-audit

The "did the loop actually do the work?" verifier. After every feature chain
runs to its last working stage, this skill walks the recorded
`stages_completed` and produces a per-session trace file recording what
evidence each stage actually generated — not just that a stage ran. The
result is the artifact `/pr-review` reads to decide whether to approve.

Source: `docs/superpowers/specs/2026-04-29-define-better-metric-and-trace-audit-design.md` §4 + §5.

## When this fires

Immediately before `gsd-ship` on every feature chain (`feature:ui`,
`feature:data`, `feature:infra`, `feature:migration`). The chain-manifest in
`contracts/chain-manifest.json` is the source of truth.

`bug` and `refactor` chains do NOT run this skill.

## Inputs

The skill reads:

- The session JSON (full record including `stages_completed`, timestamps, any
  per-stage artifact pointers).
- The pinned `metric.md` for the session (path from `metric_file`, integrity
  via `metric_sha`).
- The branch's commits since branch-off: `git log --no-merges <base>..HEAD`.
- The CI/test outputs from the most recent run (per-test pass/fail,
  screenshots, e2e Playwright traces).
- The PR diff at HEAD: `gh pr diff <pr> --patch` or the equivalent `git diff`.

## Outputs

- `docs/superpowers/traces/<session_id>.md` — the trace file matching the
  template in the next section.
- A patch to the session JSON adding `trace_file: "docs/superpowers/traces/<session_id>.md"`.

The canonical template lives in `docs/superpowers/templates/trace-template.md`.
Any future tooling that reads trace files MUST accept that shape; the inline
copy below is a copy-paste convenience.

## Deterministic walk

For each stage in `stages_completed`, the skill records three things:

1. **Evidence produced.** What file, test result, commit, or transcript
   demonstrates the stage actually ran.
2. **Move toward metric.** Did this stage's work move the system closer to the
   pinned metric, or was it overhead/setup?
   - `brainstorming`, `define-metric`, `writing-plans` → "n/a (planning)".
   - `tdd`, `executing-plans`, `e2e`, `verification-before-completion` → MUST
     have evidence of metric movement, or trace ends with
     `NEEDS-HUMAN: silent-skip-on-<stage>`.
3. **Silent skips.** Any stage where `record-stage` ran but no evidence is
   locatable in the diff or test output.

### Locator assertion rules (DETERMINISTIC — non-negotiable)

Every locator written into the trace MUST be deterministically asserted before
the row is committed. The three valid assertion forms are:

- File-system existence: `test -e <path>`
- Git object existence: `git cat-file -e <sha>`
- GitHub API existence: `gh api <pr>/files | grep -q <path>`

If a locator cannot be deterministically asserted, the row writes
`LOCATOR_UNVERIFIED` in the locator column. The trace verdict cannot be PASS
with any unverified rows.

## Metric verification

After the per-stage walk, the skill re-runs the `measurement_method` named in
the metric file. It captures the output verbatim and emits one of three
verdicts:

- `PASS` — measurement method ran clean and the result matches the `## Target`
  section of the metric file.
- `FAIL` — measurement method ran but the result does NOT match `## Target`.
- `INCONCLUSIVE` — measurement method could not be run (missing test, manual
  step not yet executed, etc.).

## Gaming pre-mortem cross-check

For each scenario the metric file enumerated under `## Gaming pre-mortem`, the
skill must locate the guard or write `NO GUARD`. Any `NO GUARD` finding
appends `NEEDS-HUMAN: ungarded-gaming-scenario` to the verdict block.

## NEEDS-HUMAN failure modes

The trace verdict is one of `PASS | NEEDS-HUMAN | FAIL`. The three
`NEEDS-HUMAN` shapes the skill emits:

- `NEEDS-HUMAN: silent-skip-on-<stage>` — a non-planning stage was recorded
  but no evidence of metric movement is locatable.
- `NEEDS-HUMAN: ungarded-gaming-scenario` — a gaming pre-mortem scenario from
  the metric file has no corresponding guard in the diff.
- `NEEDS-HUMAN: locator-unverified` — at least one row was written with
  `LOCATOR_UNVERIFIED`.

A trace ending in any `NEEDS-HUMAN` line blocks `/pr-review` approval.

## Trace file template

The skill writes the trace matching this exact shape.

```markdown
# Trace — <session_id>

**Branch:** <branch>
**Metric file:** <metric_file_path> (sha <metric_sha>)
**Generated by:** harness:trace-audit subagent (<model>) at <iso8601>

## Per-stage evidence

| Stage | Evidence kind | Locator | Moves metric? |
|-------|---------------|---------|---------------|
| brainstorming | transcript | session.brainstorming_log | n/a (planning) |
| define-metric | metric file | docs/superpowers/metrics/<id>.md | n/a (defines target) |
| writing-plans | plan file | docs/superpowers/plans/<id>.md | n/a (planning) |
| tdd | failing tests | tests/<path>::<test> at commit <sha> | yes (defines acceptance) |
| executing-plans | green tests + diff | tests passing at commit <sha>; diff: <files> | yes |
| e2e | playwright trace | .pr-review/e2e-<id>.zip | yes |
| verification-before-completion | manual click-path log | .pr-review/verify-<id>.md | yes |
| requesting-code-review | reviewer artifacts | .pr-review/review-<id>.json | yes |
| trace-audit | this file | self | n/a |

## Metric verification

**Stated metric:** <copy from metric.md>
**Measurement result at HEAD:** <ran the measurement method, output verbatim>
**Verdict:** PASS | FAIL | INCONCLUSIVE

## Gaming pre-mortem cross-check

For each gaming scenario the metric file enumerated, did the implementation guard against it?

- <scenario 1 verbatim from metric.md> → <guard at <file>:<line> | NO GUARD>
- <scenario N> → ...

If any "NO GUARD" appears: trace ends with `NEEDS-HUMAN: ungarded-gaming-scenario`.

## Silent skips

<List or "None.">

## Verdict

PASS | NEEDS-HUMAN | FAIL
```

## Authoring guidance for the trace subagent

- Walk `stages_completed` in order. For each stage, emit one row in the
  per-stage evidence table. Do NOT skip stages even if they look obvious;
  the table is the audit record.
- Assert every locator deterministically before writing the row. If you
  cannot run `test -e <path>` or `git cat-file -e <sha>` or
  `gh api <pr>/files | grep -q <path>`, write `LOCATOR_UNVERIFIED` —
  do not fabricate plausible-looking paths.
- The metric verification block re-runs the measurement method named in the
  metric file. Capture stdout verbatim. If the measurement method is manual
  ("look at the screen"), record `INCONCLUSIVE` and route to `NEEDS-HUMAN`
  with a one-liner pointing the operator at the screen.
- The gaming pre-mortem cross-check is the load-bearing audit. Each scenario
  in the metric file must map to a concrete guard at `<file>:<line>` or be
  flagged `NO GUARD`. Speculative or wishful guard-finding is a bug.

## Acceptance criteria for the skill author

- Trace committed with all five required sections in the documented order.
- Every per-stage row has a locator that has been deterministically asserted
  OR is marked `LOCATOR_UNVERIFIED`.
- Metric verification block ran the measurement method and captured its
  output verbatim.
- Gaming pre-mortem cross-check covers every scenario in the metric file.
- Verdict line is exactly one of `PASS | NEEDS-HUMAN | FAIL` and matches the
  evidence above.
